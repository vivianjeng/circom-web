§3 · The math

R1CS & Powers of Tau.

How a constraint becomes a matrix row, a set of rows becomes a polynomial, and a polynomial is verified at a secret point — without ever revealing the secret.

Every <== compiles to one R1CS row

Each constraint takes the form (A·w)(B·w) = (C·w) where w is the witness vector and A, B, C are sparse coefficient vectors — one row per constraint.

// c <== a * b  →  one R1CS constraint

// signals: [  1    a    b    c  ]

A  =  [  0    1    0    0  ]   // A·w = a
B  =  [  0    0    1    0  ]   // B·w = b
C  =  [  0    0    0    1  ]   // C·w = c

// check: a × b = c  ✓

The first element of w is always 1 (the constant signal). All other signals follow in declaration order. A circuit with n constraints = n such rows stacked into three sparse matrices.

WHY RANK-1

Each constraint is the product of two linear combinations — degree 2. "Rank-1" refers to the outer-product structure A·Bᵀ in the matrix form, which is what makes these constraints efficiently provable.

From rows to one polynomial identity (QAP)

Groth16 doesn't check n rows one by one. It converts the matrices to polynomials via Lagrange interpolation over n evaluation points (roots of unity mod p), then checks one identity:

A(x) · B(x) − C(x) = H(x) · Z(x)

A(x), B(x), C(x)

Witness-weighted sums of column polynomials: A(x) = Σ wᵢ·Aᵢ(x), same for B and C. Each Aᵢ(x) interpolates column i of matrix A across all evaluation points.

Z(x) — vanishing polynomial

Z(x) = (x−ω)(x−ω²)···(x−ωⁿ). Zero at every evaluation point. If any constraint is violated, A(x)·B(x)−C(x) is not divisible by Z(x) — no valid H(x) exists.

H(x) — quotient

The prover computes H(x) = (A(x)·B(x)−C(x)) / Z(x). A valid H(x) proves all n constraints hold simultaneously — one polynomial check instead of n row checks.

Why evaluate at one secret point?

Checking every coefficient of A(x)·B(x)−C(x) = H(x)·Z(x) would mean transmitting and verifying polynomials of degree proportional to the circuit size — far too large and slow. Instead, the verifier exploits the Schwartz-Zippel lemma: if two distinct polynomials of degree d agree at a random point, the probability of a false positive is at most d/|𝔽|, which is negligible over a large field.

THE SHORTCUT

Instead of checking the full polynomials, the verifier only checks:

A(τ) · B(τ) − C(τ)
= H(τ) · Z(τ)

One evaluation point. Constant-size proof. Same security.

WHAT IS τ

τ is a field element chosen during a multi-party ceremony, used once, then destroyed. Nobody — not the prover, not the verifier — ever learns its value after setup. What the ceremony publishes are encrypted powers:

[gτ⁰, gτ¹, gτ², …, gτⁿ]

Anyone can compute gf(τ) for any polynomial f — but nobody can extract τ.

POWER OF TAU — PHASE 1

Powers of Tau is the first phase of the trusted setup. Its only job is to produce the encrypted powers of τ in a way that no single party ever sees τ in the clear.

Phase 2 (circuit-specific setup) then uses these powers to bake the constraint structure into the proving/verifying keys.

BUT WAIT — THE PROVER DOESN'T KNOW τ EITHER

We said the verifier checks A(τ), B(τ), C(τ). But to generate those values the prover needs to evaluate:

A(τ) = a₀ + a₁·τ + a₂·τ² + a₃·τ³ + …

The coefficients aᵢ come from the witness — the prover knows those. But τ is secret and destroyed. How can you plug a number you don't know into a formula?

✗ WHAT THE PROVER CANNOT DO

a₀ + a₁ × τ + a₂ × τ² + …

Direct arithmetic on τ — impossible. τ no longer exists anywhere.

✓ WHAT THE PROVER CAN DO

a₀·[1]G + a₁·[τ]G + a₂·[τ²]G + …

Scale the published encrypted powers by their own coefficients and sum — this lands on [A(τ)]G without ever touching τ directly.

This works because scalar multiplication on an elliptic curve is linear: a₁·(τ·G) = (a₁τ)·G. Scaling and adding curve points produces the same result as evaluating the polynomial first, then scaling — so the prover can compute the right group element without knowing τ. This is the entire reason the encrypted powers exist.

WHY POWER OF TAU EXISTS

We need to evaluate polynomials at τ — but nobody can be allowed to know τ. The solution is to pre-compute and publish the powers of τ in encrypted form before the ceremony closes.

✗ RAW NUMBERS — NEVER PUBLISHED

1   τ   τ²   τ³   …   τⁿ

Giving anyone the actual values of τ, τ², … would let them forge proofs. These numbers are generated inside the ceremony and immediately destroyed — they are the "toxic waste."

✓ ENCRYPTED FORM — WHAT IS PUBLISHED

[1]G   [τ]G   [τ²]G   [τ³]G   …   [τⁿ]G

Each value is multiplied into an elliptic-curve group element. You can use these points to evaluate any polynomial at τ — but reversing the multiplication to recover τ is computationally infeasible (discrete-log hardness).

The ceremony's sole output is the second row. No participant — not even the organisers — ever holds the first row in full. The .ptau file is the encrypted row.

ATTACK — WHY τ MUST STAY SECRET

Suppose an attacker learns τ. They want the verifier to accept a false proof — i.e., they want the check to pass at τ even though the circuit constraints are not actually satisfied:

What the attacker constructs:

A*(τ) · B*(τ) − C*(τ) = 0

They simply pick arbitrary A*, B*, C* polynomials that happen to satisfy the equation at the single known point τ — trivially easy once τ is known.

But the actual constraint is violated:

A*(x) · B*(x) − C*(x) ≠ 0

The polynomials don't correspond to any valid witness. The verifier sees only the value at τ — which was forged — and is deceived.

This is why the ceremony must be run by many independent participants: a forger would need every single participant to collude (or be compromised) to learn τ. As long as one participant is honest and destroys their share, τ is gone forever.

Powers of Tau — evaluate at a secret τ

To verify the identity, the verifier needs the polynomials evaluated at a secret point τ. If τ were known, a cheating prover could construct fake A(x), B(x) that pass the check without satisfying the circuit. So τ is generated inside a multi-party ceremony — then immediately destroyed (toxic waste).

two polynomials — one evaluation point at τ
━━ p(x) · honest polynomial (derived from valid witness) ━━ q(x) · any polynomial a cheater knowing τ could forge
WHAT THE CEREMONY PUBLISHES

[τ⁰]G · [τ¹]G · [τ²]G · … · [τᵈ]G

Each entry is an elliptic-curve point [τⁱ]G = τⁱ · G — the generator G scaled by a power of τ. The superscript is a true exponent: τ¹ = τ, τ² = τ×τ, τ³ = τ×τ×τ, and so on — not a label or a multiplier index.

Because elliptic-curve discrete-log is hard, publishing τⁱ·G reveals nothing about τ itself — yet anyone can evaluate any polynomial p(x) = p₀ + p₁x + p₂x² + … at the secret point via:

p(τ)·G = p₀·[τ⁰]G + p₁·[τ¹]G + p₂·[τ²]G + …
= Σ pᵢ · (τⁱ·G)

This works because scalar multiplication on the curve is linear — you can combine the published points with the polynomial's own coefficients pᵢ to land on the right answer, with no knowledge of τ required. These points are the contents of the .ptau file.

WHAT THE VERIFIER CHECKS

Using bilinear pairings, the verifier confirms:

e(π_A, π_B) =
e([α]₁, [β]₂) · e(pub,[γ]₂) · e(π_C,[δ]₂)

Three pairing operations. τ was baked into the keys during setup — the verifier never needs it directly.
