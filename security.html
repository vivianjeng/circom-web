§7 · Security pitfalls

A passing proof can still be wrong.

Underconstrained circuits don’t crash. They cheerfully verify garbage. Four classes of bug to learn now, before the audit finds them for you. After Y. Jeng — “security holes when writing circom”

1 · Everything is mod p

SNARK_SCALAR_FIELD · bn128 21888242871839275222246405745257275088548364400416034343698204186575808495617 ~ 254 bits
DANGEROUS

A user passes in = p (which is ≡ 0):

signal input in;
in === 0;   // passes when in = p !
SAFE

Use circomlib's IsZero — it cannot be fooled.

component iz = IsZero();
iz.in  <== in;
iz.out === 1;   // out = 1 ⇔ in = 0

2 · LessThan(n) trusts you too much

DANGEROUS

You think you're comparing two uint64s. The prover passes 200-bit values. LessThan(64) happily lies.

signal num[2];
component lt = LessThan(64);
lt.in[0] <== num[0];   // could be huge
lt.in[1] <== num[1];
lt.out === 1;
SAFE

Force inputs into the bit range with Num2Bits(n) first.

component n2b[2];
component lt = LessThan(64);
for (var i = 0; i < 2; i++) {
    n2b[i] = Num2Bits(64);   // range-check
    n2b[i].in <== num[i];
    lt.in[i] <== num[i];
}
lt.out === 1;

+1 bit for arithmetic

If you compute a + b from n-bit values, the result needs n+1 bits. Dark Forest v0.3 was bitten by exactly this.

Num2Bits(254) is wrong

It accepts values above p. Use Num2Bits_strict() when you need a no-overflow range check.

LessThan(n) caps n ≤ 252

There's an assert(n <= 252) inside. Catches mistakes at compile time — write your own asserts liberally.

3 · Underconstrained public signals

If a public signal has no constraints, it's malleable.

Tornado Cash takes recipient, relayer, fee, refund as public inputs — but the circuit doesn't use them in any computation. A miner could swap the recipient without invalidating the proof.

The fix is a one-liner per signal: square it. The optimizer can't drop the constraint, and any change to the input invalidates the proof.

// from tornado-core/circuits/withdraw.circom

signal input recipient;
signal input relayer;
signal input fee;
signal input refund;

// Bind them with a quadratic constraint each.
// Squares prevent the optimizer from
// removing them. 4 constraints total.
signal recipientSquare;
signal feeSquare;
signal relayerSquare;
signal refundSquare;

recipientSquare <== recipient * recipient;
feeSquare       <== fee       * fee;
relayerSquare   <== relayer   * relayer;
refundSquare    <== refund    * refund;

4 · Use template assertions

Add assert(SUM_FIELDS <= TOTAL_FIELDS) to your template parameters. assert fires at compile time — it catches misconfiguration before deploy, costs zero constraints.

Pre-deploy checklist

Four questions to run through before you put a circuit on mainnet. Each one maps to a pitfall above.

INPUTS & RANGE
  • Every public input that enters arithmetic is range-checked with Num2Bits(n).
  • Any comparison uses LessThan(n) only after range-checking both operands.
  • No raw === 0 or === 1 on unbounded inputs — use IsZero / IsEqual.
CONSTRAINTS & SIGNALS
  • Every public signal appears in at least one <== or === — no free-floating inputs.
  • Template parameters are guarded with assert() at the top of the template.
  • Run snarkjs r1cs print and check constraint count matches expectations.
TESTING
  • Write at least one test that should fail — pass an invalid witness and confirm the proof is rejected.
  • Try boundary inputs: 0, p−1, 2ⁿ−1 for each signal.
  • Test with inputs that are one bit too wide for every LessThan / Num2Bits.
BEFORE MAINNET
  • Import only audited circomlib templates — never copy-paste partial implementations.
  • Use a circuit-specific Phase 2 ceremony; don't reuse another project's .zkey.
  • Get an independent audit. Underconstrained bugs are invisible to tests that don't know what to look for.
