Section one

A circuit is
a system of
constraints.

Not a program. Not a function. A set of polynomial equations that some witness must satisfy. Circom is just a nicer way to write them down.

§1 · SYNTAXThe three operators
Compute vs. constrain — pick deliberately05 / 24
The three operators that trip everyone up

<--   ===   <==

ASSIGN ONLY

<--

Computes a value for a signal. No constraint is added. Used for witness-only helpers.

out <-- in / 7;

⚠ Dangerous on its own — the prover can lie.

CONSTRAIN ONLY

===

Adds a constraint. Both sides must be equal in every valid witness.

out * 7 === in;

Pure equation — no assignment.

ASSIGN + CONSTRAIN

<==

The one you want 95% of the time. Computes and constrains in one stroke.

c <== a * b;

✓ Default choice. Reach for this first.

Rule of thumb  ·   If the prover could cheat by assigning a wrong value, you used <-- when you should have used <==. remember
§1 · SYNTAXTemplates & signals
multiplier.circom06 / 24
A template is the unit of reuse — like a class

Hello, Multiplier.

pragma circom 2.1.6;

template Multiplier() {
    signal input  a;
    signal input  b;
    signal output c;

    c <== a * b;
}

component main {public [a]} = Multiplier();

↑ A complete, valid Circom program.

What to notice

  • signal — an unknown in the system; can be input, output, or intermediate.
  • template — parameterized circuit; instantiated with component.
  • main — exactly one entry point per circuit.
  • public [a] — which inputs the verifier sees. Everything else is private.
  • One <== here = one R1CS constraint after compile.
§1 · SYNTAXThe R1CS shape
Why every constraint is quadratic07 / 24
Every constraint must fit one shape

(A·s) × (B·s) = (C·s)

R1CS = Rank-1 Constraint System. s is the witness vector; A, B, C are sparse coefficient vectors. Each constraint is the product of two linear combinations.

LEGAL 
c <== a * b;
y <== 3*x + 5;        // linear is fine
z <== (a+b) * (c+d);
ILLEGAL 
y <== a * b * c;      // degree 3
y <== a / b;          // not quadratic
FIX 
signal ab;
ab <== a * b;
y  <== ab * c;        // split into 2

PROVER · runs locally, holds secrets

Has private inputs (the witness). Runs the .wasm to compute every signal, then uses the .zkey to produce a proof. Heavy compute, runs once per claim.

VERIFIER · runs anywhere, public

Has only the verification key + the public inputs + the proof. Outputs true or false. Cheap, succinct — milliseconds, kilobytes.

§4 · POSEIDONImporting from circomlib
~213 constraints for Poseidon(2)13 / 24
You almost never write a hash from scratch — you import one

Hash two secrets.

// hash2.circom
pragma circom 2.1.6;

include "circomlib/circuits/poseidon.circom";

template Hash2() {
    signal input  left;
    signal input  right;
    signal output out;

    component h = Poseidon(2);
    h.inputs[0] <== left;
    h.inputs[1] <== right;
    out <== h.out;
}

component main = Hash2();

/* INPUT = { "left": "7", "right": "42" } */

What we just did

  • Poseidon(n) — a parameterized template; n is the arity (number of inputs).
  • Components are wired by assigning their input signals with <==.
  • Output is read off h.out.
  • This is the brick we'll stack into a Merkle tree on the next slide.

§5 · BUILD #2 · MERKLE MEMBERSHIP10 minutes
Prove "I'm in the set" without saying who you are14 / 24
The most useful 30 lines of Circom you’ll ever write

Merkle membership.

root
h₂ h₃
h₀ h₁ h₄ h₅
ab cd ★f gh

Public: the root.  Private: the leaf ★, plus the sibling on each level (filled) and which side it sat on.

The claim

“I know a leaf in the tree with this root, but I won’t tell you which one.”

The recipe

  1. Start with the leaf as your cursor.
  2. For each level, hash cursor with that level’s sibling.
  3. The side bit decides ordering: left/right.
  4. After depth rounds, assert cursor == public root.
§5 · MERKLEThe whole circuit
~30 lines, ~4k constraints at depth 2015 / 24
All of the cryptographic interesting-ness fits on one screen

merkle.circom

pragma circom 2.1.6;
include "circomlib/circuits/poseidon.circom";

// Pick the correct ordering for one Merkle level.
template DualMux() {
    signal input  in[2];
    signal input  s;     // 0 = leaf left, 1 = leaf right
    signal output out[2];

    s * (1 - s) === 0;       // force s ∈ {0,1}
    out[0] <== (in[1] - in[0]) * s + in[0];
    out[1] <== (in[0] - in[1]) * s + in[1];
}

template MerkleProof(depth) {
    signal input  leaf;
    signal input  pathElements[depth];
    signal input  pathIndices[depth];
    signal output root;

    component mux[depth];
    component hash[depth];
    signal    cursor[depth + 1];
    cursor[0] <== leaf;

    for (var i = 0; i < depth; i++) {
        mux[i] = DualMux();
        mux[i].in[0] <== cursor[i];
        mux[i].in[1] <== pathElements[i];
        mux[i].s     <== pathIndices[i];

        hash[i] = Poseidon(2);
        hash[i].inputs[0] <== mux[i].out[0];
        hash[i].inputs[1] <== mux[i].out[1];
        cursor[i + 1] <== hash[i].out;
    }
    root <== cursor[depth];
}

component main {public [root]} = MerkleProof(20);

Read it twice

  • DualMux — a tiny gadget. s*(1-s)===0 is the canonical “is a bit” constraint — without it, the prover could choose s = 7.
  • cursor[] — array of intermediate signals; declared once, written once.
  • for-loop — unrolled at compile time. depth is a compile-time parameter, not a runtime one.
  • Public: only root. The leaf and path stay private.
COST
depth=20
~ 4,300 constraints
prove time
~ 1 s on a laptop
§6 · BUILD #3Private membership · mixed-secret leaf
The trick that makes mixers possible16 / 24
A leaf isn’t the user — it’s a commitment to the user

Mix the secret in.

A bare-leaf membership proof shows that some leaf is in the set. Anyone watching deposits can correlate withdrawals to deposits 1-to-1.

Fix it: a leaf is the hash of a nullifier + a secret. The nullifier becomes a public single-use marker on withdraw — the secret stays hidden forever.

The two-hash idea

commitment      = Poseidon(nullifier, secret)
nullifierHash   = Poseidon(nullifier)

// On withdraw, prove:
// 1. commitment is in tree under public root
// 2. nullifierHash matches the public one
// → contract records nullifierHash → no double-spend
pragma circom 2.1.6;
include "circomlib/circuits/poseidon.circom";
include "./merkle.circom";

template Withdraw(depth) {
    // private
    signal input  nullifier;
    signal input  secret;
    signal input  pathElements[depth];
    signal input  pathIndices[depth];
    // public
    signal input  root;
    signal input  nullifierHash;

    // 1. commitment = Poseidon(nullifier, secret)
    component cm = Poseidon(2);
    cm.inputs[0] <== nullifier;
    cm.inputs[1] <== secret;

    // 2. that commitment is the leaf in the tree
    component tree = MerkleProof(depth);
    tree.leaf <== cm.out;
    for (var i = 0; i < depth; i++) {
        tree.pathElements[i] <== pathElements[i];
        tree.pathIndices[i]  <== pathIndices[i];
    }
    tree.root === root;

    // 3. the public nullifierHash matches
    component nh = Poseidon(1);
    nh.inputs[0] <== nullifier;
    nh.out === nullifierHash;
}

component main {public [root, nullifierHash]} = Withdraw(20);

§7 · SUDOKUThree rules → three constraints
Translating game rules to polynomials18 / 24
Before code: write the constraint plan

What does “valid” mean,
in equations?

RULE 1

Each cell is 1–9

For every cell value v:

(v-1)*(v-2)*(v-3)
       ...
*(v-8)*(v-9) === 0;

A degree-9 product. Use repeated squaring or a Range gadget; circomlib has helpers.

RULE 2

Solution agrees with puzzle

For every cell where the puzzle is non-zero:

puzzle[i][j] * (
  puzzle[i][j] - sol[i][j]
) === 0;

If puzzle = 0, no constraint; otherwise sol must equal puzzle.

RULE 3

Rows / cols / boxes are permutations of 1–9

The classic encoding:

// every digit d ∈ 1..9
// must appear exactly once
// in the row.  Use IsEqual
// + sum gadgets.

Reuse for cols and 3×3 boxes — same gadget, different indexing.

Public inputs puzzle[9][9]  ·  private: solution[9][9] ~ 11k constraints
§7 · SUDOKUThe skeleton
Read top-to-bottom — Rule 1, Rule 2, Rule 319 / 24
All three rules in one template

sudoku.circom

pragma circom 2.1.6;
include "circomlib/circuits/comparators.circom";

// Rule 1 — cell is in {1..9}
template InRange1to9() {
    signal input v;
    signal acc[10];
    acc[0] <== 1;
    for (var k = 1; k <= 9; k++)
        acc[k] <== acc[k-1] * (v - k);
    acc[9] === 0;
}

template Sudoku() {
    signal input puzzle[9][9];     // public
    signal input solution[9][9];   // private

    component rng[81];

    // Rules 1 + 2
    for (var i = 0; i < 9; i++) {
        for (var j = 0; j < 9; j++) {
            rng[i*9+j] = InRange1to9();
            rng[i*9+j].v <== solution[i][j];
            puzzle[i][j] * (puzzle[i][j]
                - solution[i][j]) === 0;
        }
    }
    // Rule 3 → Distinct9 over each
    // row, col, and 3×3 box.
}

component main {public [puzzle]} = Sudoku();

Distinct9 helper

// 9 distinct digits ⇒ permutation of {1..9}
template Distinct9() {
    signal input a[9];
    component eq[36];     // C(9,2) pairs
    var idx = 0;
    for (var i = 0; i < 9; i++)
      for (var j = i+1; j < 9; j++) {
        eq[idx] = IsEqual();
        eq[idx].in[0] <== a[i];
        eq[idx].in[1] <== a[j];
        eq[idx].out === 0;
        idx++;
      }
}

Wire 27 instances — 9 rows, 9 cols, 9 boxes — to constrain the whole grid.

§8 · SECURITY · 1 of 4The field is finite
Overflow & the IsZero trap21 / 24
Pitfall #1 — finite-field overflow

Everything is mod p.

SNARK_SCALAR_FIELD  ·  bn128 21888242871839275222246405745257275088548364400416034343698204186575808495617 ~ 254 bits
DANGEROUS

A user passes in = p (which is ≡ 0).

// "is this signal zero?"
signal input in;
in === 0;   // passes when in = p !
SAFE

Use circomlib’s IsZero — it cannot be fooled.

component iz = IsZero();
iz.in  <== in;
iz.out === 1;   // out = 1 ⇔ in = 0
FOLLOW-ON

The same trap hits a + b, a * b, range-proofs — anything where you assumed integers behave like integers. They’re field elements.

§8 · SECURITY · 2 of 4LessThan(n) trusts you too much
Always range-check inputs first22 / 24
Pitfall #2 — comparators don’t check input width

Bound it, then compare it.

DANGEROUS

You think you’re comparing two uint64s. The prover passes 200-bit values. LessThan(64) happily lies.

signal num[2];
component lt = LessThan(64);
lt.in[0] <== num[0];   // could be huge
lt.in[1] <== num[1];   // could be huge
lt.out === 1;
SAFE

Force inputs into the bit range with Num2Bits(n) first — its internal sum-check will reject overflow.

component n2b[2];
component lt = LessThan(64);
for (var i = 0; i < 2; i++) {
    n2b[i] = Num2Bits(64);   // range-check
    n2b[i].in <== num[i];
    lt.in[i] <== num[i];
}
lt.out === 1;

+1 bit for arithmetic

If you compute a + b from n-bit values, the result needs n+1 bits. Dark Forest v0.3 was bitten by exactly this.

Num2Bits(254) is wrong

It accepts values above p. Use Num2Bits_strict() when you specifically need a no-overflow range check.

LessThan(n) caps n ≤ 252

There’s an assert(n <= 252) inside. Catches mistakes at compile time — write your own asserts liberally.

§8 · SECURITY · 3 + 4 of 4Underconstrained circuits
The bug that ate Tornado Cash’s safety story23 / 24
Pitfall #3 — public signals that don’t do anything

If a public signal has
no constraints, it’s malleable.

Tornado Cash takes recipient, relayer, fee, refund as public inputs — but the circuit doesn’t use them in any computation. A miner could swap the recipient without invalidating the proof.

The fix is a one-liner per signal: square it. The optimizer can’t drop the constraint, and any change to the input invalidates the proof.

PITFALL #4 · template assertions

Add assert(SUM_FIELDS <= TOTAL_FIELDS) to your template parameters. assert fires at compile time — it catches misconfiguration before deploy, costs zero constraints.

// from tornado-core/circuits/withdraw.circom

signal input recipient; // not used in any compute
signal input relayer;
signal input fee;
signal input refund;

// Bind them with a quadratic constraint each.
// Squares prevent the optimizer from
// removing them. 4 constraints total.
signal recipientSquare;
signal feeSquare;
signal relayerSquare;
signal refundSquare;

recipientSquare <== recipient * recipient;
feeSquare       <== fee       * fee;
relayerSquare   <== relayer   * relayer;
refundSquare    <== refund    * refund;
