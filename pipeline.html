PROVER · runs locally, holds secrets

Has private inputs (the witness). Runs the .wasm to compute every signal, then uses the .zkey to produce a proof. Heavy compute, runs once per claim.

VERIFIER · runs anywhere, public

Has only the verification key + the public inputs + the proof. Outputs true or false. Cheap, succinct — milliseconds, kilobytes.

circom vs snarkjs — who does what

They are separate tools with a clean handoff. circom speaks the language; snarkjs speaks the cryptography.

circom · the compiler
  • Written in Rust, installed as a binary.
  • Reads .circom source — templates, signals, constraints.
  • Outputs .r1cs, .wasm, .sym.
  • Knows nothing about proving — pure compilation.
  • Re-run only when the circuit changes.
snarkjs · the toolkit
  • JavaScript library — runs in Node or the browser.
  • Reads the artefacts circom produced.
  • Handles trusted setup, witness calc, proving, verification.
  • Can export a Solidity verifier contract for on-chain use.
  • Re-run once per setup, then once per proof.
§2.1 · snarkjs step-by-step

Seven commands, explained.

Every command hands off a file to the next. Lose one and nothing downstream works. Source: iden3/snarkjs ↗

00

Write the circuit

file: circuit.circom 
pragma circom 2.0.0;

template Multiplier2() {
    signal input  a;
    signal input  b;
    signal output c;
    c <== a * b;
}

component main {public [a]} = Multiplier2();

A circuit is just a text file. Define your signals and constraints, then pass it to circom. Every subsequent step in the pipeline reads the artefacts this produces — nothing runs without it.

01

Compile the circuit

circom circuit.circom --r1cs --wasm --sym
reads: circuit.circom → writes: circuit.r1cs circuit_js/circuit.wasm circuit.sym

Parses your Circom source and emits three artefacts. The R1CS file encodes the constraint system as sparse matrices. The .wasm is a witness calculator — feed it your inputs and it outputs every signal value. The .sym maps signal indices back to names for debugging. The --sym flag is optional but helpful.

02

Phase 2 setup — tie ptau to your circuit

snarkjs groth16 setup circuit.r1cs powersOfTau28_hez_final_14.ptau circuit_0000.zkey
reads: circuit.r1cs powersOfTau28_hez_final_14.ptau → writes: circuit_0000.zkey

Combines the universal Powers of Tau file with this circuit’s R1CS to produce the initial proving key (Phase 2 of the trusted setup). The _0000 suffix signals zero contributions so far. The ptau size must be large enough: 2k ≥ constraint count. A 214 ptau covers up to ~16 k constraints — fine for demos.

03

Contribute entropy

snarkjs zkey contribute circuit_0000.zkey circuit_final.zkey --name="me" -v
reads: circuit_0000.zkey → writes: circuit_final.zkey

Each contributor folds fresh randomness into the zkey. Security holds as long as at least one contributor destroys their randomness. For production, coordinate multiple independent parties and verify the contribution chain with snarkjs zkey verify. For local development, a single contribution is fine.

04

Export the verification key

snarkjs zkey export verificationkey circuit_final.zkey verification_key.json
reads: circuit_final.zkey → writes: verification_key.json

Extracts the small, public portion of the zkey as a verification key. This file contains no secrets — publish it freely. It’s also the input to snarkjs zkey export solidityverifier, which generates a drop-in Solidity contract for on-chain verification.

05

Compute the witness

node circuit_js/generate_witness.js circuit_js/circuit.wasm input.json witness.wtns
reads: circuit.wasm input.json → writes: witness.wtns

Runs the WebAssembly witness calculator with your private inputs. The output .wtns file holds the values of every signal — public and private, plus all intermediate ones. This file stays local. The prover consumes it in the next step but never sends it to the verifier. The witness is the secret.

06

Produce the proof

snarkjs groth16 prove circuit_final.zkey witness.wtns proof.json public.json
reads: circuit_final.zkey witness.wtns → writes: proof.json public.json

The heavy cryptographic step. Uses the proving key and witness to compute a Groth16 proof — three elliptic curve points, ~800 bytes total. Also writes public.json (the public signal values) which travels to the verifier alongside the proof. Computation time scales with constraint count, not input size.

07

Verify the proof

snarkjs groth16 verify verification_key.json public.json proof.json
# [INFO]  snarkJS: OK!
reads: verification_key.json public.json proof.json → no secrets required

Runs three elliptic-curve pairing checks and prints OK! if all pass. No witness, no private inputs, no zkey — just the verification key, the proof, and the public signals. This is exactly what the Solidity verifier contract does on-chain, and it takes < 1 ms.

§2.2 · Glossary

Key terms.

Dotted-underlined terms throughout this page link back here.

R1CS
Rank-1 Constraint System. The matrix format circom compiles to. Each constraint is one equation of the form (A·w)(B·w) = (C·w), where w is the witness vector and A, B, C are sparse coefficient vectors. Groth16 operates directly on the R1CS matrices.
Witness
The complete assignment of every signal in the circuit — private inputs, public inputs, and all intermediate signals. A valid witness satisfies every R1CS constraint. The prover holds it; the verifier never sees it.
Powers of Tau
The output of a universal trusted-setup ceremony (Phase 1). A single ptau file encodes cryptographic randomness up to 2k constraints and can be reused across any circuit of that size. Iden3 runs the ceremony; the largest file supports 228 constraints.
Trusted Setup
A multi-party computation ceremony that generates the proving and verification keys. Security rests on at least one participant destroying their toxic waste (secret randomness). Phase 1 (ptau) is universal. Phase 2 (zkey) is circuit-specific and must be re-run if the circuit changes.
zkey
The circuit-specific proving key produced by combining ptau + R1CS (Phase 2 setup). The prover uses it to generate proofs. It also contains the verification key. The zkey is large (scales with constraint count) and should be kept private until setup is finalised.
Verification Key
A small JSON file extracted from the zkey. Contains only public parameters — no secrets. Anyone holding the vkey can verify proofs. It is also compiled into a Solidity contract for on-chain verification via snarkjs zkey export solidityverifier.
Groth16
A zk-SNARK proving system (Groth, 2016). Proofs are constant-size — three elliptic-curve points, ~800 bytes — regardless of circuit complexity. Verification costs three pairing operations: fast everywhere, including on-chain. Requires a circuit-specific trusted setup per circuit.
Proof
The three-group-element object (π_A, π_B, π_C) output by the prover. Combined with public inputs, it convinces any verifier — with overwhelming probability — that the prover knows a witness satisfying the circuit, without revealing what that witness is.
.wasm
The WebAssembly witness calculator emitted by circom --wasm. Given your private inputs in input.json, it computes the value of every signal in the circuit and writes them to witness.wtns. Runs in Node via generate_witness.js and stays entirely local — the output is never sent to the verifier.
.sym
A symbol map emitted by circom --sym. Maps each signal's numeric index in the R1CS back to its human-readable name (e.g. main.a, main.out). Not required for proving or verifying — useful for debugging constraint errors and reading snarkjs wtns export json output.
