§1 · Circom syntax

A circuit is
a system of constraints.

Not a program. Not a function. A set of polynomial equations that some witness must satisfy. Circom is just a nicer way to write them down.

The three operators that trip everyone up

<--·===·<==

ASSIGN ONLY

<--

Computes a value for a signal. No constraint is added. Witness-only helpers.

out <-- in / 7;

See IsZero for the safe pattern ↓

CONSTRAIN ONLY

===

Adds a constraint. Both sides must be equal in every valid witness.

out * 7 === in;

Pure equation — no assignment.

ASSIGN + CONSTRAIN

<==

The one you want 95% of the time. Computes and constrains in one stroke.

c <== a * b;

✓ Default choice. Reach for this first.

Rule of thumb · if the prover could cheat by assigning a wrong value, you used <-- when you should have used <==.

§1.1 · A first template

Hello, Multiplier.

A template is the unit of reuse — like a class. main is the entry point.

pragma circom 2.1.6;

template Multiplier() {
    signal input  a;
    signal input  b;
    signal output c;

    c <== a * b;
}

component main {public [a]} = Multiplier();

What to notice

  • signal — an unknown in the system; can be input, output, or intermediate.
  • template — parameterized circuit; instantiated with component.
  • main — exactly one entry point per circuit.
  • public [a] — which inputs the verifier sees. Everything else is private.
  • One <== here = one R1CS constraint after compile.

Every constraint must fit one shape

(A·s) × (B·s) = (C·s)

R1CS = Rank-1 Constraint System. s is the witness vector; A, B, C are sparse coefficient vectors. Each constraint is the product of two linear combinations.

LEGAL 
c <== a * b;
y <== 3*x + 5;
z <== (a+b)*(c+d);
ILLEGAL 
y <== a * b * c;  // deg 3
y <== a / b;      // not quad.
FIX 
signal ab;
ab <== a * b;
y  <== ab * c;
§1.2 · template vs function

Two kinds of reuse.

A template is a circuit blueprint — it produces signals and constraints. A function is a compile-time helper — it returns a number and generates nothing.

TEMPLATE — generates constraints 
template Bits(n) {
    signal input  in;
    signal output out[n];

    var sum = 0;
    for (var i = 0; i < n; i++) {
        out[i] <-- (in >> i) & 1;
        out[i] * (out[i] - 1) === 0;
        sum += out[i] * (2**i);
    }
    sum === in;
}

component main = Bits(8);
  • Has signal declarations — part of the witness.
  • Template parameters (n) are compile-time constants.
  • Instantiated with component — wires are connected via dot notation (b.in <== x).
  • Can include other components — this is how circomlib is composed.
FUNCTION — compile-time only 
function nbits(n) {
    var bits = 1;
    var cur  = 2;
    while (cur < n) {
        bits++;
        cur  *= 2;
    }
    return bits;
}

template RangeBits(max) {
    var n = nbits(max); // called at compile time
    signal input in;
    // use n to size arrays...
}
  • No signals, no constraints — pure computation.
  • Supports loops, conditionals, recursion.
  • Can only be called where a var or template parameter is expected — never on the right-hand side of <==.
  • Common use: computing array sizes, bit-widths, constants.
ONE-LINE RULE if it touches a signal → template  ·  if it returns a number at compile time → function
§1.3 · When <-- is the right tool

IsZero — division inside a circuit.

Division is not R1CS-legal, so you can't write inv <== 1/in. The pattern: assign with <--, then prove the assignment was correct with ===.

// circomlib/circuits/comparators.circom
// github.com/iden3/circomlib ↗

template IsZero() {
    signal input  in;
    signal output out;
    signal        inv;

    inv <-- in != 0 ? 1/in : 0;   // ① assign

    out <== -in * inv + 1;          // ② constrain
    in * out === 0;                 // ③ constrain
}

Source: iden3/circomlib comparators.circom ↗

Why three lines instead of one

  • ① <-- computes the inverse outside the constraint system — division is fine here because it's just JavaScript arithmetic, not R1CS.
  • ② <== constrains out = −in·inv + 1. If in ≠ 0, then inv = 1/in, so out = 0. If in = 0, then out = 1.
  • ③ === adds in·out = 0. Together with ②, a dishonest prover cannot set out = 1 for a non-zero in.
THE RULE

Every <-- must be followed by === constraints that make cheating impossible. Without the constraints, the prover can assign any value to inv and the verifier would never know.

in = 0

inv = 0 (by assignment)
out = −0·0 + 1 = 1 ✓
0·1 = 0 ✓

in = 5

inv = 1/5
out = −5·(1/5) + 1 = 0 ✓
5·0 = 0 ✓

cheat attempt

Prover sets in = 5, lies with inv = 0 → out = 1.
But in·out = 5·1 = 5 ≠ 0 — constraint ③ rejects. ✗
